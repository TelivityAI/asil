# Codex Review — ASIL monorepo

Generated by `codex exec` (codex-cli 0.131.0, GPT-5-class model) running read-only over the whole repository on 2026-06-17. Findings are ranked by leverage. File/line citations are Codex's own, verified against the tree at review time.

## Concise summary

Highest-leverage issues: the executor is effectively **RCE-by-design with full env inheritance**; **System A/B budget enforcement misses major LLM phases** (self-review, adversarial gate, thinker fan-out); coverage-gap tasks can generate **absolute paths the executor later rejects**; rescans create **unbounded duplicate tasks** because task IDs are random; and **bootstrap failures** (`pnpm install` / build) are logged but not treated as fatal.

---

## Findings

### 1. `critical` — Unsafe execution boundary for untrusted repos and patches
`packages/asil-improvement-loop/src/loop.ts:217` runs `pnpm install --frozen-lockfile`, `pnpm -r build`, and later `pnpm typecheck` / `pnpm test`; `packages/asil-runners/src/wiring.ts:298` (`execFile`) inherits the **full process environment**. Any target-repo postinstall/build/test script can exfiltrate `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, git credentials, or local files.
**Fix:** default install to `--ignore-scripts`; run build/test in a container or VM with a minimal env allowlist; disable outbound network for untrusted phases; separate PR-creation credentials from execution credentials.

### 2. `critical` — Cost control does not cover all LLM spend
`loop.ts:174` creates one checkpoint but only records `executeTask()` usage at `loop.ts:280`. `selfReview()` and `adversarialReview()` are **outside checkpoint accounting**, so System A can overspend silently. System B has the same shape: `packages/asil-thought-multiplier/src/papa.ts:64` fans thinkers out first and only records aggregate usage afterward at `papa.ts:83`.
**Fix:** pass checkpoints into every LLM-using phase, record after each call, and `forceCheck()` before expensive fan-out/review stages.

### 3. `high` — Coverage-gap tasks can become unexecutable (absolute paths preserved)
`profiles/typescript.ts:96` and `profiles/python.ts:102` emit coverage entry keys verbatim; `scanner.ts:214` turns them into task `filePaths`; `executor.ts:295` later rejects absolute `<<<FILE: ...>>>` paths as suspicious. `scanner.test.ts:140` even codifies the bad shape.
**Fix:** normalize coverage paths to repo-relative before task creation, ideally by giving `coverage.extract` access to `repoRoot`.

### 4. `high` — Rescans create duplicate work forever (random task identity)
`scanner.ts:304` assigns `randomUUID()`; `task-queue.ts:37` dedupes only on `task.id`. `loop.ts:121` therefore re-enqueues the same unresolved issue on every run under a new ID.
**Fix:** derive a stable fingerprint from category + normalized file paths + normalized detector payload, and dedupe on that fingerprint.

### 5. `high` — `maxAttempts` is configured but never applied
`run-a.ts:262` sets `maxAttempts: 2`, but `loop.ts:107` constructs `new TaskQueue(config.queuePath)` without forwarding it, so `task-queue.ts:31` always uses its own default.
**Fix:** `new TaskQueue(config.queuePath, { maxAttempts: config.maxAttempts })`, and surface a CLI flag.

### 6. `high` — Worktree bootstrap failures logged but not fatal
`loop.ts:221-246` — failed `pnpm install` / `pnpm -r build` only print stderr, then the loop still calls `executeTask()`, patching in a known-bad environment and producing misleading downstream failures.
**Fix:** abort the task on bootstrap failure; record a distinct infra-failure outcome.

### 7. `high` — Domain answers keyed only by normalized question text
`domain-questions.ts:75` hashes only text; lookups/writes at `:191`/`:195` use that hash alone. Two different files asking "What is the grace period?" share one answer.
**Fix:** composite key (file path + line or stable marker ID); keep text hash as metadata, not the primary key.

### 8. `high` — Analyzer belief-action-gap detector mixes unrelated tasks
`packages/asil-analyzer/src/detectors/belief-action-gap.ts:63` buckets confidence **globally** across all calls; `:123` flags any accept/reject split in the same bucket. A 0.8 accept on task A and a 0.8 reject on task B becomes a false "same-conf-opposite-valence" hit.
**Fix:** scope buckets by `taskId` (ideally also phase/reviewer role).

### 9. `medium` — Transcript capture stores raw prompts/responses with no redaction
`instrumented-callers.ts:22` records full `systemPrompt`/`userPrompt`/`responseContent`; `transcript-writer.ts:215` writes them straight to disk. Transcripts can contain proprietary code, stack traces, or secrets pulled from source.
**Fix:** redaction hooks, secret-pattern scrubbing, restrictive file permissions, explicit opt-in warning.

### 10. `medium` — Dead-code detection too shallow for an auto-PR tool
`scanner.ts:277` treats "no cross-file grep hit" as unused — misses external package consumers, dynamic imports, re-exports, framework reflection, runtime symbol lookup.
**Fix:** downgrade dead-code findings to "candidate / manual-review", or back them with language-aware symbol analysis (`tsserver`, export-map checks, Python import graphing).

### 11. `medium` — Test coverage misses several high-risk regressions
`scanner.test.ts:151` asserts absolute coverage paths instead of catching them; no regression test for stable task identity across rescans; none that `config.maxAttempts` reaches `TaskQueue`; none that bootstrap failure aborts before execution; none that review/adversarial tokens are budget-accounted.
**Fix:** integration tests for queue persistence across multiple scans and budget accounting across every LLM phase.

### 12. `medium` — Adoption / publish readiness still repo-internal
All package manifests are `private: true`; no CI workflow files. README promises more than the distribution story delivers.
**Fix:** CI (build/test/typecheck on Linux + macOS), publish metadata for reusable packages, and a hardening checklist for running ASIL against untrusted repos.

## Top 5 things to do next, in order

1. **Harden the execution sandbox:** `--ignore-scripts` on install, scrubbed env, no network for untrusted phases, separate PR-creation credentials from execution credentials.
2. **Fix budget enforcement** so every LLM/Codex call in Systems A and B is checkpointed individually.
3. **Normalize all scanner-produced file paths** to repo-relative; add regression tests for coverage-gap execution.
4. **Replace random task IDs with stable fingerprints**; wire `config.maxAttempts` through to the queue.
5. **Make bootstrap failure fatal**; tighten analyzer correctness (belief-action-gap scoping) and add transcript redaction.
